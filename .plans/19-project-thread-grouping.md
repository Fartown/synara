# 19 —— 项目内会话分组（Thread Grouping）产品方案

## 状态

产品方案，待评审决策。未开始实现。

## 背景与问题

随着自动导入和 fork 功能的落地，单个项目下的会话数量快速增长（部分项目已有几十上百条会话）。当前侧边栏只有"项目 → 会话"两级结构，带来三个问题：

1. **fork 的会话没有归属感**：为一个任务 fork 出的多个尝试/变体散落在会话列表里，只能靠父子树的缩进勉强辨认"血缘"，无法整体折叠、无法命名。
2. **同一话题的多段会话无法聚合**：用户经常就同一话题开多段会话（不一定有 fork 关系），目前没有办法把它们归到一起。
3. **列表变长后查找困难**：排序和标题搜索只能缓解，不能替代"归类"这一心智模型。

用户诉求（原话）："fork 出来的会话以及类似话题的会话，我合并到一个组里"。

## 现状（代码事实）

- 会话（thread）属于项目（project），侧边栏为"项目 → 会话"两级。
- 已有 fork 父子树：`parent_thread_id` 表达血缘，`buildProjectThreadTree`（`apps/web/src/components/Sidebar.logic.ts:766`）支持嵌套缩进 + 展开/折叠；`fork_source_thread_id`、`sidechat_source_thread_id` 记录 fork/话题来源。
- 排序三种：`updated_at`（默认）/ `created_at` / `last_reply`（`apps/web/src/appSettings.ts:81`）。
- 侧边栏顶部有标题搜索框；会话支持置顶（`is_pinned`）和 markers（高亮/下划线，非分组）。
- **不存在"命名分组"概念**——这是本方案要新增的能力。

## 核心概念：会话分组（Thread Group）

组是**项目内的命名容器**，成为侧边栏的一级实体：

```text
项目
├── 📁 组 A（3）          ← 命名、可折叠
│   ├── 会话 1（父）
│   │   └── 会话 2（fork）  ← fork 父子树在组内继续保留
│   └── 会话 3（同话题）
├── 会话 4（未分组）
└── 会话 5（未分组）
```

基本规则：

- **单归属**：一个会话最多属于一个组；"移动到另一组"即改归属。不做多对多（避免"改了 A 组里的会话，B 组里那条是不是同一个"的认知负担）。
- **组不跨项目**：组的成员必须都在同一项目内；会话被移到其他项目时自动移出组。
- **组只是组织层**：解散/删除组不会删除任何会话，会话回到未分组区。
- **fork 父子树保留**：组不替代树。组内仍按现有父子关系缩进展示；组是树外层的"话题口袋"。

### 组的属性

| 属性 | 说明 | 存储 |
| --- | --- | --- |
| id | 不可变标识 | 服务端 |
| projectId | 所属项目（不可变） | 服务端 |
| title | 名称；默认取首个成员会话的标题（截断），可重命名 | 服务端 |
| createdAt / updatedAt / deletedAt | 常规时间戳与软删 | 服务端 |
| 成员数 / 状态聚合 | 派生值，不存储 | — |
| 折叠状态 | 纯 UI 态 | 客户端本地（同现有 `expandedParentThreadIds` 模式） |

**MVP 不做组颜色**（后续可加，数据模型预留即可）。

## 自动分组规则（fork 跟随）

- **fork 时跟随进组**：fork 新会话时，若源会话在组内，新会话自动加入同组。这是最符合直觉的默认（fork 是"同一任务的不同尝试"）。
- **不自动建组**：源会话不在组内时，fork 不会自动创建组（避免每个 fork 都产生一个无名组，组爆炸比列表长更糟）。
- **不做自动话题聚类**：不根据标题/内容相似度自动合并（误合并比不合并更伤信任）。"相似话题建议"列为后续可选能力，且只做提示、不自动执行。
- 自动进组只是默认行为，用户事后可随时移出；MVP 不设全局开关（保持简单）。

## 手动操作（核心场景，用户主动"合并"）

| 操作 | 入口 | 说明 |
| --- | --- | --- |
| 新建分组 | 会话右键"移动到 → 新建分组"；项目行 ⋯ 菜单"新建分组"（先建空组再拖入） | 名称默认取首个成员会话标题 |
| 移入组 | **拖拽**：把会话拖到组头；右键菜单"移动到 → 某组" | 拖拽是"合并类似话题"最自然的动作，建议 MVP 即支持 |
| 移出组 | 拖到未分组区；右键"移出分组" | — |
| 重命名 | 组头双击 / 右键"重命名" | — |
| 解散组 | 组头右键"解散分组" | 成员回到未分组区，会话本身不受影响 |
| 删除组 | 同"解散组" | 组不持有内容，删除即解散，无二次确认（可撤销性靠操作本身低风险） |

MVP 不做多选批量移动（后续跟多选能力一起做）。

## 侧边栏交互

- **组头**：折叠箭头 + 名称 + 成员数（如 `调研 overlay（4）`）+ **状态聚合**（任一成员运行中/待批准/待输入时，组头显示对应状态点，折叠时也能感知）+ 悬停操作按钮。
- **排序**：组是**不可拆散的硬容器**，任何排序规则下组都整体参与：
  - 组的位置 = 组内会话按当前排序键的最新值（`updated_at` 排序时=组内最近更新的会话时间，`created_at`/`last_reply` 同理）。
  - 组内按同一规则排序（含 fork 树缩进规则，与现状一致）。
- **置顶**：全局置顶区只含**未分组**的置顶会话；组内置顶仅在该组内部生效（组内排最前）。规则简单、不产生"一条会话出现两次"的问题。
- **搜索**：标题搜索命中组内会话时自动展开该组并在组头显示命中数；无命中的组整体隐藏（现有过滤逻辑的延伸）。
- **折叠**：状态存客户端本地（localStorage），不同步、不占服务端模型。
- **拖拽细节**：HTML5 drag & drop；组头和未分组区为 drop target；拖拽悬停在折叠组上约 600ms 自动展开；拖入时组头高亮。

## 数据模型与同步（服务端权威，沿用事件溯源）

### 新命令（orchestration command）

- `threadGroup.create { projectId, title }`
- `threadGroup.rename { groupId, title }`
- `threadGroup.delete { groupId }`（成员的 `groupId` 全部置空）
- `thread.setGroup { threadId, groupId | null }`（批量移动=逐条派发；保持命令原子性简单）

### 投影

- 新表 `projection_thread_groups(id, project_id, title, created_at, updated_at, deleted_at)`。
- `projection_threads` 新增 `group_id` 列（SQLite 迁移，默认 NULL）。
- 壳快照（shell snapshot）携带组列表与每条线程的 `groupId`；推送复用现有 domainEvent → 投影 → 快照链路，**不需要新 WS 通道**。

### 客户端

- 组数据进 shell snapshot 驱动的现有 store；折叠/拖拽临时态纯客户端。
- 排序/树构建在 `Sidebar.logic.ts` 扩展：`buildProjectThreadTree` 之前先按组分桶，组作为排序单元参与 `sortThreadsForSidebar`。

## 边界情况

| 场景 | 行为 |
| --- | --- |
| 删除/归档组内会话 | 成员数减一；**空组保留**（用户刚建的组不应莫名消失），可手动解散 |
| 会话移到其他项目 | 自动移出组（组不跨项目） |
| fork 时源会话在组内 | 新会话自动进同组（可事后移出） |
| 导入外部会话 | 不进任何组（导入无 fork 关系） |
| 切换排序规则 | 组位置随组内最新值变化；组永不拆散 |
| 搜索命中组内会话 | 该组自动展开并显示命中数 |
| 与 markers / 置顶并存 | 互不影响，各自独立 |

## 待决策问题

1. **单归属 vs 多归属**：推荐单归属（移动语义简单、无同步歧义）。
2. **空组保留 vs 自动解散**：推荐保留，手动解散。
3. **置顶与组的关系**：推荐"全局置顶区只含未分组置顶，组内置顶仅组内有效"。
4. **fork 自动跟随**：推荐默认开启、事后可移出、不设全局开关。是否同意？
5. **拖拽是否进 MVP**：推荐进（核心场景"合并类似话题"的最自然操作），若工期紧张可降级为"右键菜单先行"。
6. **组颜色**：MVP 不做，是否接受？
